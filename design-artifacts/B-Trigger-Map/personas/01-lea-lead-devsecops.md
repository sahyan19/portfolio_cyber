# Persona 1 — Léa, Lead DevSecOps / Tech Lead

> **Rang :** PRIMAIRE — décideuse CDI
> **Objectif business :** Convertir vers entretien CDI
> **Règle de tranchée :** en cas d'arbitrage design, on tranche pour Léa.

---

## Profil

- **Âge :** 32-42 ans
- **Poste :** Lead DevSecOps, Tech Lead, ou CTO dans une scale-up / ESN (50-500 personnes)
- **Localisation :** Paris / Lyon / Bordeaux / remote-friendly EU
- **Contexte de recrutement :** son équipe a besoin d'un junior/intermédiaire DevSecOps autonome

---

## Contexte de visite

- **Source d'arrivée :** LinkedIn (suite à candidature ou message recruteur) ou GitHub (audit des repos)
- **Durée moyenne :** 60-180 secondes pour décider de shortlister
- **Device :** Desktop 70%, mobile 30% (en déplacement)
- **Moment de la journée :** matinée ou après-midi (sourcing actif)

---

## Objectifs personnels

1. Recruter quelqu'un capable de **livrer** sans 6 mois de formation
2. Trouver un profil qui maîtrise **CI/CD + sécurité** (rare combo)
3. Réduire la dette sécurité dans son équipe
4. Éviter de gâcher 3 mois sur un mauvais recrutement

---

## Frustrations

- CV qui survend, candidats incapables de débugger un pipeline en entretien
- Profils "DevOps généraliste" sans aucune notion de sécurité
- Portfolios génériques copiés/collés sans projets réels démontrables
- Profils qui listent 50 technos sans aucune profondeur

---

## Ce qu'elle veut voir sur le portfolio (en 60 secondes)

1. **Preuve technique immédiate** — repos GitHub actifs, projets cyber concrets (Wazuh, pentesting), pas du tutoriel
2. **Stack lisible** — voir d'un coup d'œil Docker / K8s / CI/CD / Ansible / SIEM
3. **Sérieux** — ton, langue, fautes d'orthographe = signal d'alarme
4. **Contact facile** — email, LinkedIn, CV PDF téléchargeable

---

## Forces motrices

### ✅ Positives (à activer)

| Code | Force | Comment l'activer |
|---|---|---|
| **L1** | Show, don't tell : preuve > déclaration | Liens GitHub réels, captures, README détaillés. Pas de "compétent en X", mais "j'ai déployé X pour Y, voici le repo" |
| **L2** | Pattern recognition tech : reconnaître un pair sérieux | Vocabulaire précis (IaC, reverse proxy, persistance volumes, hardening), choix techniques justifiés |
| **L3** | Gain de temps : valider en 60s vs 3min | Hero qui pose immédiatement le positionnement, navigation rapide |
| **L4** | Curiosité technique — *"Il sait monter un SOC ?"* | Terminal interactif comme démo de capacité (pas gadget), commandes qui révèlent du contenu vrai |
| **L5** | Signaux d'autonomie : capable de livrer seul | Projets de bout en bout (du déploiement à la prod), maturité dans les choix |

### ❌ Négatives (à neutraliser)

| Code | Friction | Comment la neutraliser |
|---|---|---|
| **L¬1** | Peur du mis-hire ("encore un junior survendu") | Liens vers vrai code en prod, ton mesuré, projets datés et contextualisés |
| **L¬2** | Allergie au stéréotype hacker matrix | Effets cyber subtils (terminal pertinent, pas explosion de néon). Pro avant fun |
| **L¬3** | Manque de temps : 30s max si rien ne capte | Hero ultra-clair : *"Ingénieur DevOps & Cybersécurité, je livre des infras sécurisées"* |
| **L¬4** | Doute sur la maintenabilité | Code visible, structure de projets propre, IaC mentionné explicitement |

---

## Triggers déclencheurs (ce qui la fait te contacter)

- ✅ Voit un projet Wazuh/SIEM bien documenté → *"ce gars sait monter un SOC"*
- ✅ Repo GitHub avec README propre + IaC → *"il pense production"*
- ✅ Stack DevSecOps cohérente, pas un patchwork
- ✅ Méthodologie expliquée (CI/CD, tests, déploiement)

## Anti-triggers (ce qui la fait fermer l'onglet)

- ❌ Trop de "stack list" sans preuve
- ❌ Fautes d'orthographe
- ❌ Effets visuels qui ralentissent la nav
- ❌ Aucun lien GitHub réel
- ❌ Ton "rebel hacker" non professionnel
